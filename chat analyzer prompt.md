# ТЗ: Веб-анализатор экспортированных чатов Telegram

## Роль и контекст

Ты — senior full-stack разработчик. Тебе нужно создать полнофункциональное веб-приложение
"Мега-анализатор чатов" — инструмент, который принимает экспортированный JSON-чат из Telegram
(`result.json`, формат стандартного экспорта Telegram Desktop) и выдаёт богатую визуальную
аналитику: структурную статистику (несемантическую) и смысловую статистику (семантическую,
на основе NLP/embeddings).

Целевой язык переписки — **русский** (учти морфологию, стоп-слова, токенизацию кириллицы).
Делай код production-grade: с обработкой ошибок, понятными сообщениями, без заглушек "TODO".

---

## Стек

- **Backend:** Python 3.11+, FastAPI, Uvicorn
- **Очередь/прогресс:** фоновая задача (BackgroundTasks или asyncio) + эндпоинт статуса,
  опрашиваемый фронтендом (polling каждые 1–2 сек) ИЛИ WebSocket — выбери WebSocket для
  плавного realtime прогресс-бара
- **NLP/семантика:**
  - `sentence-transformers`, модель **`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`**
    (маленькая, мультиязычная, хорошо работает с русским; ~120 МБ, CPU-friendly).
    Если нужна ещё легче — `cointegrated/rubert-tiny2` (русская, очень маленькая, ~45 МБ).
    Использовать её и для эмбеддингов, и для кластеризации тем.
  - `pymorphy2` или `pymorphy3` — лемматизация русских слов (для честной статистики слов:
    "делал"/"делает"/"делаю" → одна лемма "делать")
  - `nltk` или собственный список — токенизация
  - Сентимент-анализ: модель **`cointegrated/rubert-tiny-sentiment-balanced`** (маленькая,
    русскоязычная, 3 класса: negative/neutral/positive) или `blanchefort/rubert-base-cased-sentiment`
    если нужна точность повыше (но первая приоритетна по размеру)
  - Кластеризация тем: эмбеддинги + `sklearn` (KMeans или HDBSCAN) — НЕ разворачивать BERTopic
    целиком, чтобы не тащить лишние тяжёлые зависимости, если не требуется; но если агенту
    проще — можно использовать `bertopic` с тем же sentence-transformer как backend
- **Хранение прогресса:** in-memory dict с task_id (UUID) достаточно, БД не нужна
- **Frontend:** один HTML файл с современным JS (vanilla JS или React через CDN, на твой выбор)
  + библиотека графиков **Chart.js** или **Plotly.js** (предпочтительно Plotly — у него из
  коробки есть heatmap и красивые интерактивные графики)
- **Деплой:** всё должно работать локально через `uvicorn main:app`, без внешних API-ключей,
  без облачных зависимостей. Все модели грузятся локально через HuggingFace (один раз при
  первом запуске, с кэшированием).

## Парсинг входных данных

Входной файл — `result.json` из Telegram Desktop (Export chat history). Структура:

```json
{
  "name": "Имя чата",
  "type": "personal_chat",
  "id": 123456789,
  "messages": [
    {
      "id": 1,
      "type": "message",
      "date": "2024-01-15T10:23:45",
      "from": "Иван Иванов",
      "from_id": "user123",
      "text": "Привет, как дела?",
      "text_entities": [...]
    },
    ...
  ]
}
```

Важные нюансы парсинга, которые нужно обработать:

- `text` может быть **строкой** ИЛИ **массивом объектов** (если есть форматирование, ссылки,
  упоминания) — нужно склеивать массив в плоский текст через поле `text` каждого элемента
- Сообщения без текста (только медиа): `type: "message"`, но есть поля `photo`, `file`,
  `media_type` ("voice_message", "video_message", "sticker", "animation" и т.д.) — это нужно
  для статистики по типам контента (п.12, 24)
  - `media_type: "voice_message"` → голосовое
  - `media_type: "video_message"` → видеосообщение (кругляш)
  - `photo` (поле есть) → фото
  - `media_type: "sticker"` → стикер
  - есть отдельные сервисные сообщения `type: "service"` (например, изменение названия группы,
    закреп) — их нужно отделять от обычных и не учитывать в текстовой статистике, но можно
    подсчитать отдельно
- Реакции — в новых экспортах есть поле `reactions: [{type, emoji/count, recent}]` — если
  присутствует, использовать для п.24
- Пересланные сообщения — поле `forwarded_from` — для п.23
- Отредактированные — поле `edited` (дата редактирования) — для п.16
- Если `from_id` повторяется у нескольких `from` (смена имени в Telegram) — группировать по
  `from_id`, а не по `from`, и брать последнее встреченное имя для отображения

Сделай отдельный модуль `parser.py` с функцией `parse_telegram_export(filepath) -> pd.DataFrame`,
возвращающей нормализованный DataFrame с колонками минимум:
`message_id, datetime, author_id, author_name, text, clean_text, message_type, media_type,
is_forwarded, is_edited, reactions_count, char_count, word_count`

## Предобработка текста (критично для честности статистики слов)

Перед любой статистикой по словам (топ слова, n-граммы, словарь пары) нужно:

1. Привести к нижнему регистру
2. Убрать пунктуацию и эмодзи из текстового потока (но сохранить emoji отдельно для п.30–31,
   их нужно извлекать ДО зачистки текста, через регулярку Unicode-диапазонов emoji)
3. Токенизировать
4. **Лемматизировать** каждое слово (pymorphy3)
5. **Отфильтровать стоп-слова и слова-паразиты.** Сделай расширяемый список, минимум включающий:
   - Местоимения: я, ты, он, она, мы, вы, они, меня, тебя, его, её, нас, вас, их, мне, тебе,
     ему, ей, нам, вам, им, мной, тобой, им, ими
   - Частицы/междометия/мусор: да, нет, ну, же, бы, ли, вот, так, как, а, и, но, или, то, это,
     этот, эта, эти, тот, та, те, там, тут, здесь, очень, просто, как-то, вообще, типа, короче,
     блин, ладно, окей, ок, угу, ага, не, ни
   - Смех/эмоции текстом: хах(а+), хех(е+), лол, кек, ахах(а+), ору, рофл — делать через regex
     `^ха(ха)+$`, `^а+ха+$` и т.д., а не статичный список, т.к. вариаций "хахахаха" бесконечно
   - Слова-паразиты разговорной речи: короче, типа, как бы, в общем, собственно, на самом деле,
     в принципе, скажем так
   - Стандартный русский стоп-лист NLTK (`nltk.corpus.stopwords.words('russian')`) как база +
     твой кастомный список сверху
   - **Важно:** сделать этот список редактируемым константой в начале файла `stopwords.py`,
     чтобы пользователь мог расширить при необходимости
6. Результат — колонка `clean_tokens` (список лемм без стоп-слов) для статистики слов,
   но `text`/`clean_text` (исходный, только без эмодзи) — сохранить для сентимент-анализа и
   эмбеддингов (там стоп-слова не убираем, модели сами разберутся, выкидывание слов вредит
   качеству эмбеддинга)

## Полный список статистик для реализации (50 пунктов)

Раздели backend-логику на два модуля: `stats_structural.py` (без NLP, быстрые) и
`stats_semantic.py` (требуют моделей, медленные — именно на них прогресс-бар должен
показывать основную долю времени).

### A. Структурные (несемантические) — 1–25

1. Общее количество сообщений по каждому участнику (+ % от общего)
2. Доля сообщений каждого участника (pie chart)
3. Среднее время ответа между собеседниками (медиана и среднее, без выбросов >12 часов —
   считать как "новую сессию", не как задержку ответа)
4. Самый быстрый и самый медленный ответ (с указанием даты и текста сообщений)
5. Распределение сообщений по часам суток (0–23), отдельно по каждому участнику
6. Распределение по дням недели
7. Heatmap активности: час × день недели (общая и/или раздельная по участникам — toggle)
8. Самый длинный "монолог" (макс. число подряд сообщений одного автора без ответа другого)
9. Средняя длина сообщения (символы и слова) по каждому участнику
10. Самое длинное и самое короткое сообщение (с текстом, если не media-only)
11. Количество однословных сообщений (% от общего) по каждому
12. Соотношение типов контента: текст/фото/видео/голосовые/видеосообщения/стикеры/файлы/GIF
    (stacked bar или donut)
13. Самый длинный "стрик" — подряд идущих дней с активностью
14. Самый длинный перерыв в общении (даты начала/конца паузы)
15. Количество удалённых сообщений — **ВАЖНО:** Telegram-экспорт штатно не фиксирует удалённые
    сообщения за пределами факта экспорта на момент выгрузки. Реализуй эвристику: пропуски в
    последовательности `message_id` (id идут не подряд) = индикатор удалённых сообщений,
    с явной пометкой в UI "оценка по разрывам ID, не точное число"
16. Количество отредактированных сообщений (поле `edited`) по каждому автору
17. День с максимальным количеством сообщений (пик активности) — дата + число + что обсуждали
    (топ-3 темы по кластерам из семантического блока, если готово)
18. Динамика количества сообщений по месяцам/годам (line chart, можно по каждому участнику)
19. Кто чаще пишет первым после паузы >2 часов (% инициации диалога) — по каждому участнику
20. Количество сообщений с "!" и с "?" (отдельно), также % от всех сообщений автора
21. Среднее число сообщений в "сессии" (сессия = сообщения с разрывом <15 минут друг от друга)
22. Гистограмма распределения длины сообщений (по символам), общая и по авторам
23. Количество пересланных сообщений (`forwarded_from`) по автору
24. Использование reactions (если есть в JSON) — топ эмодзи-реакций и кто их ставит/получает чаще
25. Соотношение "длинных" (>200 симв.) vs "коротких" (<20 симв.) сообщений по автору

### B. Семантические (на NLP/embeddings) — 26–50

26. Топ-20 самых частых слов (лемм, после фильтрации стоп-слов) по каждому участнику — bar chart
27. Word cloud (облако слов) по всей переписке и по каждому участнику отдельно
28. Sentiment-анализ каждого сообщения → line chart изменения среднего сентимента по неделям/месяцам
29. Самые "позитивные" и самые "негативные" дни (агрегация sentiment по дням, топ-5 каждого)
30. Топ-15 эмодзи (по частоте) общий и по каждому участнику
31. Категоризация эмодзи по смыслу (любовь ❤️😘, смех 😂🤣, грусть 😢😭, удивление 😱😮) —
    простой словарь-маппинг emoji→категория, посчитать долю каждой категории по автору
32. Частота мата/слов-паразитов разговорной речи (используя тот же стоп-словарь паразитов —
    отдельный счётчик "сколько раз употреблено", а не просто фильтрация) по автору
33. Тематическое моделирование: кластеризация сообщений по эмбеддингам (KMeans, 8–12 кластеров
    по умолчанию, число — параметр), для каждого кластера — топ ключевых слов через TF-IDF
    внутри кластера + 2–3 примера сообщений
34. Динамика тем во времени — какие кластеры доминировали в каждый период (stacked area chart
    по месяцам, доля каждой темы)
35. Lexical diversity (TTR — type-token ratio: уникальные леммы / всего лемм) по каждому автору —
    кто богаче на язык
36. "Фирменные фразочки" — биграммы/слова, частота которых у автора А значимо выше, чем у
    автора Б (можно через log-odds ratio или просто разницу нормализованных частот) — топ-10
    на каждого
37. Частота упоминания имён собственных (NER, можно эвристикой: капитализированные слова не в
    начале предложения + не в стоп-листе имён дней/месяцев, или через `natasha`/`spacy ru_core`
    если хочется точнее) — топ-10 упоминаемых людей
38. Извлечение упоминаний дат/событий — поиск паттернов типа "завтра", "в субботу", дат в формате
    ДД.ММ, названий месяцев — список найденных "планов" с контекстом (просто список в UI,
    не на графике)
39. Соотношение вопросов к утверждениям по автору (% сообщений с "?")
40. Частота извинений — поиск лемм "извинить", "простить", "виноват" — счётчик по автору
41. Частота "люблю"/комплиментов — лемма "любить" + словарь комплиментарных слов (красивый,
    умный, молодец, классный, лучший и т.п.) — счётчик по автору
42. Индекс формальности — доля слов из разговорного/сленгового словаря vs нейтральная лексика
    (простая эвристика: список типичного слова-сленга "норм", "чё", "щас", "спс" и т.д. —
    % таких слов в речи автора)
43. Динамика "интимности" эмодзи по времени — доля категории "любовь" (из п.31) по месяцам,
    line chart — растёт/падает близость в выбранной метрике
44. Топ-10 сообщений с самым низким (негативным) sentiment score — потенциальные конфликтные
    моменты, вывести списком с датой и контекстом (±1 сообщение до/после)
45. "Переломные точки" — моменты, где скользящее среднее sentiment резко падает или растёт
    (детект через производную скользящей средней, порог — параметр) — отметить на графике
    sentiment timeline (п.28) вертикальными линиями
46. Простая эвристика "иронии/шуток": сообщения с эмодзи 😂🤣 рядом с негативным по содержанию
    текстом, или с маркерами "ладно бы", "ну такое" — % "вероятно шуточных" сообщений по автору
    (помечать явно как грубую эвристику, не точный детектор)
47. Топ-15 биграмм и топ-15 триграмм (после леммизации и фильтрации стоп-слов) — общие и по
    автору
48. Доля сообщений CAPS LOCK (>60% букв в верхнем регистре при длине >3 символов), доля
    сообщений без единого знака пунктуации, средний % ошибок/опечаток (эвристика: слова не
    найденные в словаре pymorphy и не являющиеся именами/слова из словаря разговорного сленга) —
    по автору
49. Эмоциональная синхронность — корреляция Пирсона между sentiment-рядами двух участников
    (агрегированными по дням), скользящее окно — показывает, "заражаются" ли собеседники
    настроением друг друга
50. "Словарь пары" — N-граммы (1-3 слова), которые встречаются в этом чате намного чаще, чем
    в общеязыковой частоте (грубая прокся: слова с частотой в чате выше порога, но отсутствующие
    или редкие в стандартном частотном словаре русского языка — если нет внешнего частотного
    словаря под рукой, можно просто взять самые частые n-граммы, которые НЕ являются типичными
    общими фразами — то есть слова длиннее 4 букв с частотой топ-30 по чату) — список "внутренних
    приколов/жаргона" пары

> Для пунктов, где нужна качественная оценка "хорошо/плохо это сделано" — делай рабочую
> эвристику и явно лейбли в UI как "оценочно" / "примерная метрика", чтобы не создавать
> ложного впечатления точности там, где это статистическая прикидка.

## Прогресс-бар обработки

Это ключевое требование. Реализация:

1. Пользователь загружает `result.json` через `<input type="file">` на фронтенде
2. POST на `/api/upload` → backend сохраняет файл, создаёт `task_id`, запускает обработку в
   фоне (BackgroundTasks или `asyncio.create_task`), сразу возвращает `task_id`
3. Обработка разбита на явные стадии с весами (для реалистичного % прогресса, а не "тающего
   прогресс-бара"):
   - Парсинг JSON — 5%
   - Очистка/токенизация/лемматизация текста — 15%
   - Структурная статистика (пункты 1–25) — 10%
   - Генерация эмбеддингов для всех сообщений (самая долгая часть для большого чата) — 40%
   - Sentiment-анализ всех сообщений — 15%
   - Кластеризация тем + остальная семантика (26–50) — 15%
4. Backend на каждой стадии обновляет in-memory словарь:
   `{task_id: {"stage": "Генерация эмбеддингов...", "progress": 47, "eta_seconds": 38}}`
5. ETA считать по простой формуле: время, потраченное на уже пройденные стадии / их вес,
   экстраполировать на оставшиеся — не нужно точное, но должно быть разумным, а не статичной
   заглушкой
6. Эндпоинт `/api/status/{task_id}` — отдаёт текущий статус. Используй **WebSocket**
   (`/ws/status/{task_id}`) и пушь обновления каждые 0.5–1 сек, фронтенд просто рисует
   полоску прогресса и текст текущей стадии + примерное время до конца
7. По завершении — backend кладёт результат (JSON со всеми посчитанными метриками и данными
   для графиков) в тот же словарь под `result`, фронтенд получает сигнал `"stage": "done"` и
   запрашивает `/api/result/{task_id}`, дальше рендерит дашборд
8. Обработать кейс ошибки (битый JSON, неверный формат, пустой чат) — отдельный статус `"error"`
   с понятным сообщением для пользователя, без падения сервера

## Фронтенд / UI

Создай один современный, красивый дашборд (`index.html`, можно с inline `<style>` и `<script>`,
или React через CDN — на твой выбор, но предпочтение **vanilla JS + Plotly.js**, чтобы не тащить
билд-систему).

Требования к UI:

- **Экран загрузки:** drag-and-drop зона для `result.json` + кнопка выбора файла. Красивая,
  с понятной подсказкой "куда брать файл" (короткая инструкция: Telegram Desktop → чат → три
  точки → Export chat history → выбрать JSON формат)
- **Экран прогресса:** анимированный прогресс-бар (0–100%), текст текущей стадии
  ("Анализирую тональность сообщений..."), примерное время до конца, можно добавить лёгкую
  анимацию/иконку, чтобы не было ощущения, что страница зависла
- **Дашборд результатов:** структурируй по секциям, повторяющим A/B разделение выше:
  - Шапка с общими цифрами (общее число сообщений, период переписки, участники, средн.
    время ответа) — крупные "карточки"-метрики
  - Раздел "Активность" — heatmap, timeline по месяцам, часы/дни недели
  - Раздел "Кто здесь главный" — доли сообщений, инициация диалога, длина сообщений, стрики
  - Раздел "Слова и язык" — топ слова (bar chart), word cloud, биграммы/триграммы, lexical
    diversity, формальность речи
  - Раздел "Эмоции" — sentiment timeline с отметками "переломных точек", топ
    позитивные/негативные дни, эмодзи-категории, синхронность настроения
  - Раздел "Темы разговоров" — визуализация кластеров (можно как treemap или просто карточки
    с топ-словами кластера + примерами), динамика тем по времени
  - Раздел "Забавные находки" — словарь пары, фирменные фразочки, топ конфликтных моментов,
    самые длинные монологи — подавать как "интересные находки", это самый "вирусный"/шерообразный
    раздел, оформить наиболее живо
- Тёмная и светлая тема (toggle), адаптивная вёрстка (мобильный/десктоп)
- Все графики — интерактивные (hover-тултипы с деталями, zoom где применимо через Plotly)
- Кнопка "Скачать отчёт" — экспорт в PDF или хотя бы в standalone HTML со встроенными данными
  графиков (опционально, если время позволяет — не блокирующее требование)
- Указать явно в UI: **обработка происходит локально на сервере, никакие данные не уходят
  на сторонние серверы/API** (важно, т.к. чат может содержать личную переписку) — добавить
  это как текст на экране загрузки, чтобы пользователь не переживал за приватность

Используй гайд по дизайну (если в среде есть skill frontend-design — свериться с ним перед
вёрсткой) для выбора шрифтов, цветовой палитры, отступов — избегай шаблонного вида
"бутстрап по умолчанию", сделай так, чтобы дашборд выглядел как продукт, а не как учебный пример.

## Дополнительные технические требования

- Код должен работать на чатах от пары сотен до ~100 000+ сообщений без падения по памяти —
  использовать `pandas` эффективно (батчевая генерация эмбеддингов через
  `model.encode(texts, batch_size=64, show_progress_bar=False)`, не по одному сообщению в цикле)
  и обновлять прогресс между батчами
- Для очень больших чатов — кластеризацию (KMeans) и эмбеддинги выполнять на сэмпле, если
  сообщений > 50 000, явно указать пользователю "анализ тем основан на выборке N сообщений из
  M для скорости"
- requirements.txt должен быть точным и проверенным (без несовместимых версий между
  `sentence-transformers`, `transformers`, `torch`)
- README с понятной инструкцией запуска (`pip install -r requirements.txt`, `uvicorn main:app
  --reload`, открыть `index.html` или раздать его статикой через FastAPI `StaticFiles`)
- Используй `torch` CPU-версию по умолчанию (большинству пользователей не нужен GPU для
  моделей такого размера) — но не блокируй использование GPU, если он есть
- Не используй платные API (OpenAI, Anthropic API и т.д.) — всё должно работать офлайн после
  скачивания моделей с HuggingFace при первом запуске
