# ТЗ #6: Поддержка WhatsApp-экспортов (.txt) наряду с Telegram

## Контекст

Приложение сейчас умеет парсить только Telegram-экспорт (`result.json`). Нужно добавить
поддержку второго формата — экспорт чата из WhatsApp, который выгружается как обычный `.txt`
файл через "Settings → Chats → Export chat" (с медиа или без).

Это не замена, а **дополнение**: приложение должно само определять, какой файл загружен
(Telegram JSON или WhatsApp TXT), и обрабатывать его соответствующим парсером, но дальше —
весь остальной пайплайн (50+ метрик, BERTopic, эмоции, прогресс-бар, дашборд) должен работать
**одинаково** независимо от источника, через единый нормализованный формат данных.

Это итерация поверх существующего кода. Не переписывай существующий Telegram-парсер — он
рабочий, добавляй параллельный путь для WhatsApp.

## Формат WhatsApp-экспорта

WhatsApp выгружает обычный текстовый файл, формат строки **зависит от локали телефона**, два
основных варианта, которые нужно поддержать:

**Формат с квадратными скобками (iOS, многие локали):**
```
[15.01.2024, 10:23:45] Иван Иванов: Привет, как дела?
[15.01.2024, 10:24:12] Мария: Привет! Все хорошо, у тебя как?
[15.01.2024, 10:25:01] Иван Иванов: ‎<Прикрепленный файл: IMG_001.jpg>
```

**Формат без скобок, с дефисом (Android, многие локали):**
```
15.01.2024, 10:23 - Иван Иванов: Привет, как дела?
15.01.2024, 10:24 - Мария: Привет! Все хорошо, у тебя как?
```

Нюансы, которые обязательно нужно обработать:

1. **Многострочные сообщения** — если сообщение содержит перевод строки (`\n`), последующие
   строки **не начинаются** с метки даты/времени — их нужно склеивать с предыдущим сообщением,
   а не считать новыми сообщениями без автора
2. **Системные сообщения** (без автора) — типа `15.01.2024, 10:00 - Мария создала группу
   "Поездка"`, `‎Сообщения и звонки защищены сквозным шифрованием`, `‎Иван изменил название
   группы` — определяются по **отсутствию `: ` после имени** (нет паттерна "Автор: текст",
   просто "текст события") — выносить в отдельный список service-сообщений, аналогично тому,
   как уже сделано для Telegram `type: "service"`
3. **Медиа-плейсхолдеры** — WhatsApp без экспорта медиафайлов вставляет текст-заглушку вместо
   реального файла, конкретная формулировка зависит от языка приложения:
   - `<Media omitted>` (английская локаль)
   - `‎<Прикрепленный файл: ИМЯ_ФАЙЛА>` (русская локаль)
   - `audio omitted`, `video omitted`, `image omitted`, `Voice call`, `Missed voice call` и т.п.
   Нужно распознавать эти паттерны и заполнять `media_type` соответствующим значением
   (`photo`, `video`, `voice_message`, `call`), а не считать их обычным текстом — заведи список
   regex-паттернов для основных языковых вариантов (минимум русский и английский), сделай
   список легко расширяемым (отдельная константа в коде)
4. **Невидимые/служебные unicode-символы** — WhatsApp часто вставляет `\u200e` (LEFT-TO-RIGHT
   MARK) перед строками — обязательно зачищать при парсинге, иначе он будет мешать
   регуляркам и появляться в текстах сообщений
5. **Удалённые сообщения** — WhatsApp при удалении сообщения отправителем оставляет в экспорте
   текст вида "Это сообщение удалено" / "This message was deleted" — распознавать как
   `is_deleted_placeholder: true`, не как обычный текст, и не включать в текстовую/семантическую
   статистику (но можно посчитать как отдельную метрику "сообщений удалено отправителем" —
   это даже точнее, чем эвристика по разрывам ID, которая использовалась для Telegram)
6. **Редактирование** — в отличие от Telegram, стандартный WhatsApp-экспорт **не отмечает**
   отредактированные сообщения никак (WhatsApp в принципе разрешил редактирование сравнительно
   недавно и это слабо отражается в экспорте) — если редактирование не определяется, оставь
   метрику "Отредактировано" равной 0 для WhatsApp-источника и явно отметь это ограничение в
   UI (например, маленькая подсказка/иконка "?" возле метрики: "WhatsApp не сохраняет историю
   правок в экспорте")
7. **Форматирование дат** — может быть `ДД.MM.ГГГГ` или `MM/ДД/ГГГГ` в зависимости от локали
   устройства — определять автоматически по эвристике (если день >12 — точно ДД.MM, если
   неоднозначно — пробовать ДД.MM.ГГГГ как приоритетный вариант для русскоязычного контекста,
   но сделать это настраиваемым параметром на случай ошибки автоопределения)
8. **12-часовой формат времени** — некоторые локали используют `10:23 PM` вместо `22:23` —
   обработать оба варианта

## Архитектурное решение

### 1. Определение формата файла

В `main.py` (эндпоинт `/api/upload`) добавь определение типа загруженного файла:
- Расширение `.json` → пробовать парсить как Telegram-экспорт (с валидацией структуры —
  проверить наличие ключей `messages`, `name`/`id` на верхнем уровне; если не подходит —
  понятная ошибка "Файл не похож на стандартный экспорт Telegram")
- Расширение `.txt` или `.zip` (WhatsApp иногда экспортируется в zip с медиафайлами — для
  начала достаточно поддержать сам текстовый файл, медиафайлы из zip не нужно анализировать,
  только сам `_chat.txt`/аналогичный файл внутри) → парсить как WhatsApp
- Если зашёл `.zip` — автоматически найти внутри `.txt` файл с историей чата и распарсить
  именно его, остальное содержимое архива игнорировать

### 2. Новый модуль `parser_whatsapp.py`

Создай функцию `parse_whatsapp_export(filepath) -> pd.DataFrame`, которая возвращает
**ровно такую же структуру DataFrame**, как существующий `parse_telegram_export()`:
`message_id, datetime, author_id, author_name, text, clean_text, message_type, media_type,
is_forwarded, is_edited, reactions_count, char_count, word_count`

Где данных нет в WhatsApp-формате (например, `reactions_count`, `is_edited`, `is_forwarded`)
— заполняй разумными дефолтами (`0`, `False`, `None`) и **не ломай** остальной пайплайн,
который ожидает эти колонки.

`message_id` — у WhatsApp нет встроенных ID сообщений, сгенерируй последовательный целочисленный
ID по порядку чтения файла (это означает, что эвристика "удалённые по разрывам ID" из
Telegram-пайплайна **не применима** к WhatsApp — для WhatsApp используй вместо неё прямой
подсчёт явных "это сообщение удалено" плейсхолдеров, см. пункт 5 выше; сделай так, чтобы
backend знал источник данных (`source: "telegram" | "whatsapp"`) и выбирал подходящую
эвристику/формулировку для метрики "удалённые сообщения" в зависимости от источника)

`author_id` — у WhatsApp нет стабильного ID автора (только отображаемое имя из телефонной
книги или номер телефона, если контакт не сохранён) — используй само имя автора как
`author_id` (с нормализацией: trim, нижний регистр для сравнения, но сохраняй оригинальный
регистр для отображения), это достаточно надёжно в пределах одного экспорта.

### 3. Общий слой после парсинга

Убедись, что после получения DataFrame (из любого источника) дальше идёт **полностью общий**
код — `preprocessing.py`, `stats_structural.py`, `stats_semantic.py` не должны знать, из
какого источника пришли данные, кроме тех мест, где это явно нужно (формулировка метрики
удалённых сообщений, наличие/отсутствие данных о редактировании — такие места явно
помечай комментарием `# WhatsApp/Telegram-specific` в коде).

### 4. UI: обновить экран загрузки

1. Обнови инструкцию на экране загрузки — добавь короткие инструкции для обоих источников:
   - "**Telegram:** Desktop-приложение → чат → ⋮ → Export chat history → выбрать формат JSON"
   - "**WhatsApp:** Открыть чат → имя контакта/группы → Export Chat → (без медиа быстрее)"
2. Drag-and-drop зона должна принимать `.json`, `.txt`, `.zip` (укажи это в `accept` атрибуте
   input и в текстовой подсказке)
3. После успешной загрузки — показать определённый источник ("Обнаружен формат: WhatsApp") до
   начала обработки, чтобы пользователь мог сразу понять, что файл прочитан верно

### 5. Групповые чаты — что нужно проверить параллельно

Раз уж трогаем парсер — оба формата (Telegram и WhatsApp) поддерживают групповые чаты с N
участниками. Поскольку среди существующих метрик есть штуки, изначально рассчитанные на
ровно двух собеседников ("Инициация диалога", "Кто закрывает разговор", "Эмоциональный
отклик/эмпатия", парные cosine-similarity метрики типа "похожесть стиля А vs Б"), сделай:

1. Если `len(unique_authors) == 2` — показывай все парные метрики как сейчас
2. Если `len(unique_authors) > 2` — адаптируй:
   - "Инициация диалога" — % сообщений каждого автора, открывающих новую сессию (после паузы),
     просто список по каждому автору вместо сравнения "А vs Б"
   - "Кто закрывает разговор" — аналогично, список по каждому автору
   - "Эмоциональный отклик" — для группы это становится матрицей N×N (отклик каждого на
     каждого) — если время позволяет, выведи как heatmap; если не успеваешь — на первой
     итерации **скрывай** эту метрику для групп >2 участников, чем показывай ломаную/непонятную
     визуализацию (лучше явный "Эта метрика доступна только для чатов с 2 участниками", чем
     путаница)
   - "Похожесть стиля" (cosine similarity между авторами) — для группы выведи как
     матрицу/heatmap N×N либо как просто текстовый список топ-3 самых похожих пар, в
     зависимости от того, что проще тебе реализовать в рамках этой итерации
   - Цветовая палитра графиков (donut chart "Доля сообщений", и т.д.) — убедись, что при N>2
     участниках используется палитра, которая остаётся читаемой хотя бы до 8-10 авторов
     (используй стандартную качественную Plotly-палитру типа `Plotly.colors.qualitative.Set3`
     вместо ручных 2-3 цветов, прописанных под пару)
3. Не нужно специальной логики для определения "это группа или личный чат" заранее — просто
   пусть код реагирует на фактическое количество уникальных авторов после парсинга

## Проверка перед сдачей

1. Создай (или попроси у пользователя) тестовый WhatsApp `.txt` экспорт на русском и/или
   английском языке, прогони через парсер, убедись:
   - Многострочные сообщения склеиваются правильно, не разбиваются на отдельные "пустые"
     сообщения без автора
   - Системные сообщения не попадают в список участников чата
   - Медиа-плейсхолдеры корректно размечены как `media_type`, не как обычный текст со словами
     "Media omitted" в статистике слов
   - Даты и время распознаны верно (сверь вручную пару записей с оригинальным файлом)
2. Прогони тот же существующий тестовый Telegram-чат — убедись, что ничего не сломалось от
   добавления нового парсера (регрессионная проверка)
3. Если есть доступ к тестовому групповому чату (Telegram или WhatsApp, 3+ участника) —
   прогони и проверь, что дашборд не падает и не показывает мусорные/нечитаемые графики на
   большом числе участников
4. Явно перечисли в ответе, какие метрики были адаптированы для групп, а какие временно
   скрыты для >2 участников с объяснением почему
