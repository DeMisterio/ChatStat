# ТЗ #10: Режим "Advanced Analysis" — поддержка тяжёлых LLM (Gemma 4 31B и аналоги)

## Контекст

До сих пор LLM в проекте использовалась только для одной узкой задачи — короткая подпись
названия темы после BERTopic (один промпт на кластер, 2-4 слова на выходе). Это специально
рассчитано на лёгкие модели (`qwen3:4b`, `gemma3:4b`).

Теперь у части пользователей есть доступ к по-настоящему мощным локальным моделям (например,
**Gemma 4 31B** — 24GB+ VRAM, топ-3 открытая модель в мире по бенчмаркам, доступна через
Ollama) — либо на десктопе с RTX 3090/4090, либо через арендованный GPU (Vast.ai и подобные).
Нужно добавить **отдельный, явно опциональный режим "Advanced Analysis"**, который использует
мощность такой модели не только для подписи тем, но для качественно более глубокого анализа
всей переписки — того, что маленькие модели физически не могут сделать хорошо (синтез,
интерпретация, долгий контекст).

Это не замена существующего пайплайна — это **надстройка**, включаемая отдельным тоглом,
которая использует тот факт, что мощная модель тянет на порядок больше контекста и качества
рассуждения, чем 4B-модели.

Это итерация поверх существующего кода. Не трогай базовый пайплайн (BERTopic, эмбеддинги,
структурные метрики) — добавляй новый опциональный слой поверх готового результата.

## Принцип режима Advanced Analysis

1. Новый toggle в "Расширенных настройках": **"Advanced Analysis (требует мощную локальную
   модель, рекомендуется Gemma 4 31B / Qwen 2.5 32B+)"** — выключен по умолчанию (как и текущий
   toggle LLM-подписей тем, который остаётся отдельной, более лёгкой опцией для слабого железа)
2. Когда включён — после завершения **всего** обычного пайплайна (все 80 метрик, темы,
   эмоции, и т.д.) запускается **дополнительная стадия**, которая прогоняет уже готовые
   агрегированные данные (не сырые сообщения целиком — это бы не влезло в контекст и было бы
   избыточно дорого) через выбранную мощную модель, чтобы получить более качественные,
   синтезированные результаты в местах, где маленькие модели или чисто эвристические подходы
   справлялись плохо.
3. **Список конкретных улучшений в Advanced Analysis** (что выгодно делегировать мощной
   модели вместо эвристик/маленьких моделей):

   a. **Качественные названия тем с объяснением** — вместо короткого "Обсуждение футбола"
      (как делает лёгкая модель сейчас), мощная модель получает топ-15 ключевых слов кластера
      + 5-8 примеров сообщений + размер кластера, и выдаёт: короткое название (как раньше) +
      **одно-два предложения описания**, что это была за тема и какой в целом был тон
      обсуждения. Формат ответа — строгий JSON (`{"title": "...", "description": "..."}"`),
      чтобы парсить надёжно.

   b. **Общее эссе-резюме о переписке** (новый раздел дашборда — "Резюме от AI", явно
      помеченный как генерируемый LLM, не как объективная метрика). Подели на вход модели:
      топ-темы, их пропорции, эмоциональный профиль каждого автора (CEDR-радар), ключевые
      структурные метрики (кто инициирует, кто закрывает, длина монологов), топ-фразы. Попроси
      модель написать связный текст на 150-250 слов о характере этой переписки и отношений
      между участниками — **на основе только присланных агрегированных данных**, не сырых
      сообщений (это и экономит контекст, и более этично — не нужно прогонять через модель
      личные сообщения целиком, если можно обойтись агрегатами). Добавь явный disclaimer под
      этим разделом: "Это интерпретация AI на основе статистики, не объективный факт. Могут
      быть неточности."

   c. **Улучшенная детекция "переломных моментов"** (п.45/Баг #6) — вместо чисто
      статистической эвристики (резкая производная скользящего среднего sentiment), отправь
      модели короткие текстовые окна вокруг каждого статистически найденного "кандидата в
      переломный момент" (10-15 сообщений до/после) и попроси её **подтвердить или
      отклонить**, действительно ли там видно реальное изменение в отношениях/тоне (а не,
      например, общую шутку с резким матом, которая статистически выглядит как "обрыв
      sentiment", но по смыслу не является переломным моментом). Модель возвращает
      `{"is_real_turning_point": bool, "reason": "..."}"` на каждый кандидат — отфильтруй
      ложные срабатывания эвристики.

   d. **Глубокая проверка "самых ярких моментов"** (Баг #15 — "доброе сообщение") — после
      того как эвристика (CEDR + cosine similarity) предложила топ-5 кандидатов, дай мощной
      модели весь топ-5 с контекстом (окружающие сообщения) и попроси **выбрать одно лучшее**
      с коротким объяснением почему — это тот случай, где маленькая модель/чистая эвристика
      регулярно ошибается (как мы видели на практике с "пошёл работать"), а мощная модель с
      reasoning-режимом (Gemma 4 поддерживает configurable thinking mode) должна справляться
      ощутимо лучше, потому что может реально понять контекст и иронию.

   e. **Связный нарратив "эмоциональных арок"** — вместо разметки периода одним словом
      ("Нейтрально"/"Грусть" и т.д.), дай модели сводку по эмоциям каждого периода + примеры
      сообщений, и попроси сформулировать **короткую фразу-заголовок периода** (как в самом
      начале хотелось — "период восторга и знакомства", "период бытовой рутины") — это
      явно то, для чего нужен LLM-synthesis, а не чистая статистика.

4. **Технические детали реализации:**
   - Все вызовы Advanced Analysis идут через тот же Ollama REST API
     (`http://localhost:11434/api/generate`), просто с другой моделью, выбранной в
     отдельном dropdown (не путать с dropdown для лёгких подписей тем — Advanced Analysis
     имеет свой собственный выбор модели, с рекомендацией `gemma4:31b` по умолчанию **если**
     она найдена в `/api/tags`, иначе предлагать пользователю `ollama pull gemma4:31b`)
   - Поддержи **configurable thinking mode** Gemma 4 явно — судя по документации модели, это
     управляется параметром в `apply_chat_template`/в самом запросе (`enable_thinking: true`).
     Для задач (c) и (d) выше (проверка переломных моментов, выбор лучшего доброго сообщения)
     — включай thinking mode, это задачи, требующие реального рассуждения, не короткой
     генерации. Для (a) и (e) (короткие подписи/заголовки) — можно оставить thinking mode
     выключенным для скорости.
   - Так как Advanced Analysis работает **после** основного пайплайна на агрегированных
     данных (не на тысячах сырых сообщений), общее время выполнения этой стадии должно быть
     умеренным даже с мощной моделью — счётчик запросов: один на тему (обычно 5-15 тем), один
     на общее резюме, по одному на каждый кандидат переломного момента (обычно <10), один на
     выбор лучшего доброго/тёплого/напряжённого момента, один на каждую эмоциональную арку
     (обычно 8-15 периодов) — суммарно порядка 30-60 запросов к модели, не тысячи
   - Добавь отдельный вес в прогресс-бар для стадии "Advanced Analysis" (~10-15% сверху от
     базового пайплайна), с понятным текстом стадии — "Углублённый анализ с помощью Gemma 4..."
   - Если Advanced Analysis включён, но Ollama/выбранная модель недоступна в момент запуска —
     **не валить всю обработку в ошибку**: откатиться на обычный результат без Advanced слоя,
     показать уведомление в UI ("Advanced Analysis недоступен: модель не найдена. Показан
     обычный результат.") — тот же принцип graceful degradation, что и в ТЗ#3 для HF-токена

5. **UI размещение:**
   - Toggle "Advanced Analysis" — в Расширенных настройках, рядом с существующим toggle для
     лёгких LLM-подписей тем (можно сделать их взаимоисключающими через radio-выбор "Без LLM" /
     "Лёгкие подписи (быстро)" / "Advanced Analysis (требует мощную модель, медленнее, но
     заметно глубже)" — так понятнее иерархия опций, чем два независимых чекбокса)
   - Новый раздел дашборда **"Резюме от AI"** — в самом верху дашборда, сразу под шапкой с
     карточками метрик, с explicit disclaimer про интерпретацию модели
   - Описания тем (п. a) — добавляются как дополнительный текст в уже существующих карточках
     кластеров, под названием темы
   - Подтверждённые/отклонённые переломные моменты — обнови существующий sentiment timeline
     (из Бага #6), оставляя на графике только те вертикальные линии, что подтверждены моделью
     (или показывай обе категории разным цветом — подтверждённые ярче, отклонённые приглушённо,
     дай пользователю обе версии увидеть)
   - Заголовки эмоциональных арок — заменяют текущие плоские "Нейтрально"/"Грусть" подписи на
     карточках (из Бага #16) сгенерированной фразой, но **оставляй базовую эмоцию рядом**
     (например, карточка показывает: "Период бытовой рутины 😐" с маленькой подписью "доминанта:
     нейтрально" под фразой) — так пользователь видит и сырые данные, и интерпретацию

## Проверка перед сдачей

1. Включи Advanced Analysis с любой доступной мощной моделью (Gemma 4 31B, если она
   установлена и пользователь готов её гонять, либо любая другая модель 20B+, доступная для
   теста) — убедись, что весь дополнительный слой (a-e) реально генерируется и корректно
   парсится (особенно строгий JSON-формат для подписей тем — добавь fallback на случай, если
   модель вернёт невалидный JSON: попробуй re-parse/regex-извлечение, если не получилось —
   используй старое короткое название без описания, не ломай весь раздел)
2. Выключи Ollama специально перед тестовым запуском с включённым Advanced Analysis —
   убедись, что пайплайн откатывается на обычный результат без падения
3. Засеки реальное время выполнения стадии Advanced Analysis на тестовом чате — если время
   неожиданно большое (мощные модели на CPU/слабом GPU могут быть ощутимо медленнее на каждый
   отдельный запрос, даже если их немного) — задокументируй реальное наблюдаемое время в
   комментарии кода, чтобы скорректировать вес прогресс-бара под реальность, а не предположение
4. Прочитай глазами 3-5 сгенерированных описаний тем и общее резюме переписки — убедись, что
   они релевантны входным данным, не выглядят как общие шаблонные фразы без связи с конкретным
   чатом
