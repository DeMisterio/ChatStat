## БАГ #1 (КРИТИЧНЫЙ): Стоп-слова фильтруются не везде

### Симптом
В разделе "Топ популярных слов" (структурная статистика) фильтрация работает — там леммы
типа "брат", "мочь", "знать", "ладный". Но в разделе "Темы разговоров" (тематическая
кластеризация, п.33/34) тег-облака кластеров показывают чистый мусор: `да, не, нет, что, это`,
`меня, мне, на, ты, что`, `как, на, не, что, это` — то есть стоп-слова там не фильтруются
вообще.

### Причина
Очевидно, что есть две разные точки токенизации в кодовой базе: одна (правильная, со
стоп-словами) используется в `stats_structural.py` для топ-слов, вторая (без фильтрации)
используется внутри тематического модуля для построения TF-IDF-тегов кластера.

### Требуемое исправление
1. Сделай **одну единственную функцию** `get_clean_tokens(text: str) -> list[str]` в
   `preprocessing.py`, которая делает: токенизация → лемматизация (pymorphy3) → удаление
   пунктуации/эмодзи → фильтрация по стоп-листу (`stopwords.py`) → фильтрация regex для
   смеховых вариаций (`^ха(ха)+$` и т.п.)
2. Эта функция должна вызываться **везде**, где угодно нужны "чистые слова для статистики":
   топ-слова (26), word cloud (27), TF-IDF теги кластеров (33), биграммы/триграммы (47),
   "словарь пары" (50), lexical diversity (35), "фирменные фразочки" (36)
3. Сделай ревизию кода: найди ВСЕ места, где вызывается токенизация/сплит текста на слова
   для статистических целей, и замени на вызов `get_clean_tokens`. Если где-то была
   собственная мини-токенизация внутри функции кластеризации — удали её и используй общую.
4. **Тест на регрессию:** после фикса прогони пересчёт на том же чате и убедись, что ни в
   одном из топов слов/тегов кластеров не встречаются: да, нет, не, что, это, на, ты, я, он,
   она, мы, вы, меня, мне, тебе, как, так, ну, бл (без контекста), ладно — выведи в консоль
   при тестировании список топ-20 слов каждого кластера и глазами проверь на мусор перед тем,
   как считать задачу выполненной

## БАГ #2 (КРИТИЧНЫЙ): Слабое тематическое моделирование — апгрейд архитектуры

### Симптом
Даже после фикса стоп-слов, темы представлены только как мешок ключевых слов
(TF-IDF-теги + 3 примера сообщений). Это даёт слабое читаемое представление: пользователь
должен сам додумывать, о чём кластер, по словам типа "футбол, мяч, гол".

### Требуемое решение — полная замена пайплайна тем (п.33, 34, 17)

**Шаг 1 — Лучший эмбеддер для русского языка.**
Замени текущую модель эмбеддингов для задачи тематического моделирования (можно оставить
лёгкую модель для остальной семантики, если так быстрее, но для ТЕМ используй отдельную,
более качественную):
- Основная: **`sentence-transformers/paraphrase-multilingual-mpnet-base-v2`** (мультиязычная,
  заметно лучше качество кластеризации чем MiniLM, ~420 МБ, на CPU/M3 не критично медленнее)
- Альтернатива (если хочется максимум качества под русский): **`ai-forever/sbert_large_nlu_ru`**
  или **`intfloat/multilingual-e5-large`** (e5 модели требуют префикс `"query: "` перед текстом
  при энкодинге — учти это в коде)
- Сделай выбор модели для тематики параметром в конфиге (`config.py` или `.env`), не хардкодь —
  пользователь должен мочь переключить на более лёгкую, если процессор слабый

**Шаг 2 — Полноценный BERTopic вместо ручного KMeans+TF-IDF.**
Замени текущую реализацию кластеризации на `bertopic`:
```python
from bertopic import BERTopic
from sentence_transformers import SentenceTransformer
from sklearn.feature_extraction.text import CountVectorizer

embedding_model = SentenceTransformer("paraphrase-multilingual-mpnet-base-v2")

# КРИТИЧНО: CountVectorizer для генерации ключевых слов темы должен получать
# на вход уже очищенные тексты (после get_clean_tokens, склеенные обратно в строку),
# чтобы внутренний механизм tf-idf BERTopic не подбирал стоп-слова повторно
vectorizer_model = CountVectorizer(
    ngram_range=(1, 2),
    stop_words=get_full_stopwords_list(),  # передать наш кастомный список явно
    min_df=3
)

topic_model = BERTopic(
    embedding_model=embedding_model,
    vectorizer_model=vectorizer_model,
    language="multilingual",
    min_topic_size=max(15, len(messages) // 200),  # адаптивный размер темы под объём чата
    calculate_probabilities=False,  # для скорости на больших чатах
    verbose=True
)
topics, probs = topic_model.fit_transform(cleaned_texts, embeddings=precomputed_embeddings)
```
- Используй уже посчитанные эмбеддинги сообщений (из общего пайплайна), не считай дважды
- BERTopic сам выделит "нерелевантный" кластер `-1` (outliers) — это нормально, в UI назвать
  его "Разное / несвязанные сообщения" и не показывать как полноценную тему, либо показывать
  последним и менее заметно
- Параметр `nr_topics="auto"` или фиксированное число через `nr_topics=N` — дай выбор в UI
  (слайдер "сколько тем выделить", 5–20, по умолчанию auto)
- После фита — `topic_model.reduce_outliers()` для распределения части outlier-сообщений по
  ближайшим темам, если outliers >30% от чата (иначе темы будут пустыми и неинформативными)

**Шаг 3 — Человекочитаемые названия тем через локальную LLM (опционально, переключаемо).**

Добавь **отдельный, явно переключаемый** в UI режим "Умные названия тем" (toggle/checkbox на
экране загрузки или в настройках перед запуском анализа):

- **Если выключен (по умолчанию для слабых машин):** название темы = топ-3 слова из
  BERTopic-репрезентации, через слэш — например "футбол / гол / матч". Без LLM, быстро.
- **Если включён:** после того как BERTopic выделил кластеры и их ключевые слова + примеры
  сообщений, отправить **один короткий промпт на кластер** в локальную LLM через **Ollama API**
  (`http://localhost:11434/api/generate` или `/api/chat`, стандартный Ollama REST), который
  возвращает короткое (2-4 слова) человекочитаемое название темы на русском.

  Промпт для LLM (используй такой шаблон в коде):
  ```
  Вот ключевые слова и примеры сообщений из одной темы переписки в мессенджере:
  Ключевые слова: {top_keywords}
  Примеры сообщений:
  1. {example_1}
  2. {example_2}
  3. {example_3}

  Придумай короткое название этой темы (2-4 слова, на русском, как заголовок).
  Ответь ТОЛЬКО названием, без пояснений и кавычек.
  ```

  - Реализуй через `httpx`/`requests` к Ollama REST API — **никакого внешнего интернет-API**,
    только локальный Ollama на машине пользователя
  - **Модель по умолчанию: `qwen2.5:1.5b`** (самая лёгкая адекватная вариант, отлично идёт на
    M3 MacBook). Дать пользователю выбор модели через dropdown во фронтенде — список моделей
    нужно получать динамически через `GET http://localhost:11434/api/tags` (список того, что
    уже скачано у пользователя в Ollama), а не хардкодить
  - Если Ollama не запущен/не установлен — **не ломать приложение**: ловить ошибку соединения,
    показать в UI понятное сообщение "Ollama не найден — используются названия по ключевым
    словам. Установите Ollama (ссылка) и запустите `ollama pull qwen2.5:1.5b`, чтобы включить
    умные названия", и автоматически откатываться на режим топ-слов
  - На каждый кластер — один короткий запрос (не на каждое сообщение!), это должно добавлять
    буквально секунды на десяток тем, а не минуты
  - В прогресс-баре добавь отдельную подстадию "Генерация названий тем через LLM..." если
    режим включён, с своим весом в общем прогрессе (~3-5%)

**Шаг 4 — UI для настройки тематического анализа.**
На экране загрузки (или в свёрнутом блоке "Расширенные настройки") добавь:
- Toggle "Использовать локальную LLM для названий тем (через Ollama)" — выключен по умолчанию
- При включении — dropdown с моделями, подтянутыми из `/api/tags` Ollama, плюс кнопка
  "Проверить подключение к Ollama" (зелёная/красная индикация статуса)
- Слайдер "Количество тем" (5–20, или чекбокс "Авто")
- Dropdown "Качество эмбеддингов для тем": Быстро (MiniLM) / Качественно (mpnet, по умолчанию)
- Сохрани выбор в localStorage браузера (не в backend — это просто настройка интерфейса),
  чтобы при повторном использовании не перенастраивать каждый раз

**Шаг 5 — Динамика тем во времени (п.34) после апгрейда.**
Пересчитать с новыми BERTopic-темами: для каждого месяца — доля сообщений, относящихся к
каждой теме (`topic_model.topics_over_time()` — у BERTopic есть встроенный метод именно для
этого, используй его вместо ручной агрегации, он также учитывает динамику самих топиков).

## БАГ #3: "System" как полноценный участник чата

### Симптом
В списке участников чата фигурирует "System" с 0% сообщений, 0 символов средней длины — это
служебные/сервисные сообщения Telegram (смена названия группы, закреп, добавление участника),
которые согласно изначальному ТЗ должны были полностью исключаться из текстовой статистики.

### Исправление
1. В `parser.py` убедись, что сообщения с `type: "service"` **полностью отделяются** на этапе
   парсинга в отдельный DataFrame/список (`service_messages`), который **не участвует** ни в
   одном из 50 показателей про "участников"
2. Если `from_id`/`from` сервисных сообщений равен `null`/`None`/отсутствует — проверь, что
   именно отсюда берётся фантомный "System"-участник, и явно фильтруй `author_id is not None`
   при построении списка реальных участников чата для дропдаунов/легенд графиков
3. Сервисные сообщения можно показать отдельно как маленькую справочную секцию ("За время
   переписки: переименований чата — N, новых участников — M" и т.д.), но не мешать их в
   основную статистику по авторам

## БАГ #4: "Средний ответ: 0 мин" и "Самый быстрый ответ: 0 сек"

### Симптом
Карточка на главном экране показывает медианное/среднее время ответа как "0 мин", и "самый
быстрый ответ" — 0 секунд с текстом "Понял". Это похоже на баг, а не реальный паттерн.

### Вероятная причина
Вероятно, в расчёт времени ответа попадают: (а) сообщения от того же автора подряд
(не считается "ответом" по смыслу, должно идти в п.21 "сессия", не в п.3/4), и/или
(б) сгруппированные альбомы медиа (несколько фото, отправленных одним действием, в Telegram
экспортируются как отдельные сообщения с одинаковым/почти одинаковым timestamp) — это даёт
кучу "ответов за 0 секунд", которые на самом деле не ответы, а технические дубли.

### Исправление
1. При расчёте "времени ответа" (п.3, 4) учитывай **только пары сообщений, где автор сменился**
   (`author_id[i] != author_id[i-1]`) — это уже должно было быть так по ТЗ, проверь, что
   реализовано именно так в коде, не просто "разница между соседними сообщениями"
2. Дополнительно — схлопни сгруппированные медиа-альбомы: если несколько сообщений одного
   автора идут с разницей <2 секунд и имеют `grouped_id` (в Telegram-экспорте у альбомов фото
   общий `grouped_id` в JSON) — считай их одним "сообщением" для целей расчёта времени ответа
   и инициации диалога, иначе они искажают много метрик (длину монологов, среднюю длину и т.д.)
3. Округление в UI — если медиана действительно меньше 60 секунд (быстрая переписка), показывай
   в секундах, а не "0 мин" (форматирование: <60 сек → "Xs", <60 мин → "Xm", иначе "Xh Ym") —
   так пользователь увидит "47 сек", а не обескураживающий "0 мин"
4. Для "самого быстрого ответа" — добавь минимальный порог исключения: не считать ответом
   разницу <3 секунд между разными авторами тоже (физически невозможно прочитать и осознанно
   ответить за 1-2 сек, скорее всего это тоже технический артефакт пересылки/реакции) —
   пометь это как настраиваемый порог в конфиге, не хардкодь магическое число без объяснения

## БАГ #5: Удалённых сообщений "оценка" (14192 из 39520 — 36%) неправдоподобна

### Симптом
Эвристика "разрывы в message_id ⇒ удалённые сообщения" дала 36% от всего чата — это
статистически крайне маловероятно для обычной личной переписки.

### Вероятная причина
В реальных Telegram-экспортах разрывы ID возникают НЕ только из-за удалений, но и из-за:
- Сервисных сообщений, которые могут учитываться в нумерации ID, но отфильтровываться при
  парсинге раньше, чем считается разрыв
- Медиа-альбомов (один альбом из 5 фото может занимать ID по-другому, чем кажется)
- Реакций и других не-message событий, которые в некоторых версиях экспорта тоже расходуют ID
- Если в коде разрыв считается ПОСЛЕ удаления сервисных сообщений из DataFrame — естественно,
  каждое отфильтрованное сервисное сообщение даёт фантомный "разрыв", который интерпретируется
  как удаление

### Исправление
1. Считай разрывы ID **до** любой фильтрации (на полном списке `messages` из исходного JSON,
   включая service-сообщения), и только после этого вычитай из количества "пропущенных ID"
   те, что объясняются законными service-сообщениями (которые тоже имеют свой `id` в JSON)
2. Формула: `пропущенные_id = (max_id - min_id + 1) - количество_всех_объектов_в_messages`
   (где "всех" — это и message, и service, и любые другие типы из исходного JSON, до всякой
   бизнес-фильтрации)
3. Если после этого исправления цифра всё ещё выглядит большой (>10% чата) — это, возможно,
   законное поведение Telegram ID (не строго последовательны по дизайну в некоторых случаях),
   и нужно **смягчить формулировку в UI**: вместо "Удалённых сообщений (оценка): N" писать
   "Возможных удалённых сообщений: до N (грубая оценка по ID, может включать иные причины
   пропусков)" — честность важнее красивой цифры
4. Добавь в README/код комментарий, явно объясняющий ограничения этой эвристики
