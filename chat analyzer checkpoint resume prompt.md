# ТЗ #7: Checkpoint-система — возобновление прерванной обработки

## Контекст

Обработка одного чата занимает ~10 минут (эмбеддинги, BERTopic, sentiment, эмоции, LLM-подписи
тем). Если сервер падает, перезапускается, или пользователь случайно закрывает терминал
посередине обработки — весь прогресс сейчас теряется, и нужно начинать с нуля. На слабом
железе (8GB unified memory, без дискретной GPU) это особенно болезненно — повторный прогон
того же чата стоит реального времени.

Нужна система, которая сохраняет промежуточные результаты каждой стадии пайплайна на диск,
и при повторной обработке того же файла — подхватывает с последней успешно завершённой
стадии, а не считает всё заново.

Это итерация поверх существующего кода (после ТЗ #1-#6). Правь точечно `progress_tracker.py`
и точки вызова стадий в `main.py`/пайплайне, не переписывай всё с нуля.

---

## Принцип работы

1. При загрузке файла вычисляется **хэш содержимого** (`hashlib.sha256` от байтов файла) —
   это служит стабильным идентификатором "этого конкретного чата", независимо от того, как он
   называется или когда был загружен повторно
2. На диске создаётся директория `backend/.checkpoints/{file_hash}/`, куда после **каждой**
   завершённой стадии пайплайна сохраняется промежуточный результат в виде файла (parquet для
   DataFrame — компактнее и быстрее, чем CSV; pickle/json для python-объектов типа списков
   эмбеддингов, готового BERTopic-объекта и т.д.)
3. Стадии и что сохраняется на каждой (соответствует весам из `progress_tracker.py`,
   установленным в ТЗ #1):
   - `01_parsed.parquet` — после парсинга JSON/TXT в DataFrame
   - `02_preprocessed.parquet` — после очистки/токенизации/лемматизации (добавлены колонки
     `clean_text`, `clean_tokens`)
   - `03_structural_stats.json` — результат всех структурных метрик (1-25 + новые из ТЗ#5)
   - `04_embeddings.npy` — массив эмбеддингов сообщений (numpy, сохранять через `np.save`,
     компактный бинарный формат)
   - `05_sentiment_emotions.parquet` — результаты sentiment + CEDR emotion-классификации,
     добавленные как новые колонки к DataFrame
   - `06_topics.pkl` — обученная модель BERTopic (у неё есть встроенный метод
     `topic_model.save(path)`/`BERTopic.load(path)`, использовать его, не голый pickle)
   - `07_topic_labels.json` — названия тем (от LLM или топ-слов), отдельно от самой модели,
     т.к. это быстро пересчитывается и может захотеться переделать независимо от тяжёлого
     BERTopic-фита
   - `08_semantic_extras.json` — результаты cosine-similarity метрик (забота, благодарность,
     флирт, юмор, похожесть стиля), детектор юмора, эхо-фразы и т.д. из ТЗ#5
   - `09_final_result.json` — финальный собранный JSON, который сейчас отдаётся фронтенду
     целиком по готовности (то, что лежит в `result` в текущем in-memory task store)
4. **Манифест прогресса** — `manifest.json` в той же папке, обновляется после каждой стадии:
   ```json
   {
     "file_hash": "...",
     "source_filename": "result.json",
     "source_type": "telegram",
     "stages_completed": ["01_parsed", "02_preprocessed", "03_structural_stats"],
     "last_updated": "2026-06-26T14:32:00",
     "config_snapshot": {
       "use_llm_topic_labels": true,
       "ollama_model": "qwen3:4b",
       "topic_embedding_model": "paraphrase-multilingual-mpnet-base-v2",
       "n_topics": "auto"
     }
   }
   ```
   `config_snapshot` важен — если пользователь поменял настройки (например, включил LLM-подписи
   тем, которые раньше были выключены) между запусками, нужно знать, что стадии 06/07 нужно
   пересчитать заново, даже если файл `06_topics.pkl` физически на диске есть (см. логику
   инвалидации ниже)

## Логика возобновления

В `main.py`, в начале обработки (`/api/upload` → запуск фоновой задачи):

1. Посчитать хэш загруженного файла
2. Проверить, существует ли `backend/.checkpoints/{file_hash}/manifest.json`
3. Если существует:
   - Прочитать `manifest.json`
   - **Сравнить `config_snapshot`** с текущими настройками, выбранными пользователем в UI
     (модель эмбеддингов для тем, включена ли LLM, число тем, и т.п. — все параметры,
     которые меняют поведение стадий 04 и далее)
   - Для каждой стадии в `stages_completed` — если соответствующий параметр конфига не
     менялся **и** все стадии до неё (и включая её) тоже не затронуты изменением конфига —
     можно пропустить пересчёт этой стадии, загрузить готовый файл с диска
   - Если конфиг изменился в части, которая влияет на стадию N — стадии **от N и дальше**
     нужно пересчитать заново (предыдущие, не зависящие от изменённого параметра, можно
     оставить закешированными)
   - Простое правило для упрощения логики: раздели зависимость стадий по конфигу на группы —
     стадии 01-03 (парсинг, препроцессинг, структурная стата) не зависят ни от каких
     ML-настроек, всегда можно использовать кэш, если он есть. Стадии 04-05 (эмбеддинги,
     sentiment/эмоции) зависят от выбора модели эмбеддингов/sentiment — если эти настройки не
     менялись, использовать кэш. Стадии 06-08 (темы, подписи, семантические extras) зависят от
     настроек тем (model, n_topics, llm on/off) — пересчитывать при изменении этих настроек
4. Если кэш отсутствует или невалиден для какой-то стадии — выполнить её и **сразу** сохранить
   результат на диск перед переходом к следующей стадии (не ждать конца всего пайплайна, чтобы
   частичный прогресс не терялся при повторном обрыве)
5. Обновлять WebSocket-прогресс с учётом того, что часть стадий была мгновенно загружена из
   кэша — показывать это пользователю явно в UI ("Стадии 1-3 загружены из кэша" с быстрым
   прыжком прогресс-бара, не обманывать пользователя видом, будто всё считается с нуля)

## UI: что видит пользователь

1. При загрузке файла, который уже был обработан ранее (хэш совпал) — показать модальное
   окно/баннер: **"Этот чат уже был проанализирован ранее (дата последней обработки: ...).
   Использовать сохранённый результат или пересчитать заново?"** с двумя кнопками:
   - "Показать сохранённый результат" — мгновенно грузит `09_final_result.json`, без
     прохождения пайплайна вообще
   - "Пересчитать" — запускает обработку заново, но **использует кэш для неизменных стадий**
     (см. логику выше), что всё равно быстрее полного прогона, если поменялась только пара
     настроек в конце пайплайна
2. Если обработка прерывается (сервер падает посередине, пользователь видит ошибку
   соединения с WebSocket) — при повторной загрузке **того же файла** через тот же интерфейс,
   пайплайн должен автоматически продолжить с последней сохранённой стадии, без необходимости
   вручную выбирать что-либо — это и есть "возобновление с того места, где прервалось"

## Очистка старых checkpoint'ов

Чтобы директория `.checkpoints` не росла бесконечно при анализе многих разных чатов:
1. При старте сервера — удалять checkpoint-папки старше 30 дней (configurable константа)
2. Добавить эндпоинт `DELETE /api/checkpoints/{file_hash}` и кнопку в UI "Удалить сохранённый
   анализ этого чата" — на случай, если пользователь хочет освободить место вручную
3. В README — упомянуть, что данные переписки хранятся локально в `.checkpoints` до тех пор,
   пока не удалены, это важно для приватности (это не противоречит уже заявленному в ТЗ#1
   принципу "всё локально, ничего не уходит на сторонние серверы" — но раз данные теперь
   персистентны на диске, а не только in-memory во время обработки, стоит явно это
   проговорить пользователю)

## Проверка перед сдачей

1. Загрузи тестовый чат, дай обработке дойти примерно до середины (например, до окончания
   эмбеддингов), вручную останови сервер (`Ctrl+C`)
2. Перезапусти сервер, загрузи **тот же файл** ещё раз — убедись, что:
   - Стадии 1-4 (парсинг, препроцессинг, структурная стата, эмбеддинги) не пересчитываются,
     подхватываются мгновенно из кэша
   - Обработка продолжается с sentiment/эмоций и далее
   - Итоговый результат идентичен (или очень близок — для LLM-подписей тем возможна
     небольшая вариативность между запусками) тому, что получился бы при полном прогоне без
     прерывания
3. Загрузи тот же файл третий раз без изменений настроек — убедись, что показывается
   баннер "уже проанализирован" и результат грузится мгновенно из `09_final_result.json`
4. Поменяй настройку (например, включи LLM-подписи тем, если до этого было выключено),
   загрузи тот же файл — убедись, что стадии 1-5 взялись из кэша, а 6-8 пересчитались с новой
   настройкой
5. Убедись, что обработка **другого** файла (другой хэш) не путается с кэшем первого чата —
   запускается полностью независимый пайплайн в отдельной checkpoint-папке
