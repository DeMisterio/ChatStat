# ТЗ #11: Шифрование переписок для безопасной работы на арендованных GPU (Vast.ai)

## Контекст

Часть пайплайна (в первую очередь — Advanced Analysis из ТЗ#10 с тяжёлыми моделями типа
Gemma 4 31B) предполагается запускать на арендованных GPU-инстансах (Vast.ai и подобные
маркетплейсы). Это значит, что сырые личные переписки физически будут передаваться на чужое
оборудование, которым управляет неизвестный хост-провайдер.

Нужен модуль шифрования: файл чата шифруется **локально, на машине пользователя**, перед
загрузкой на удалённый инстанс. На инстансе он расшифровывается непосредственно перед
обработкой (ключ передаётся туда же, тем же каналом, каким приходит зашифрованный файл —
это сознательный архитектурный выбор пользователя, не предполагающий защиты от
самого оператора арендованной машины, см. честное предупреждение ниже).

Модуль должен быть **исключён из git** (`.gitignore`) в части, где хранятся ключи/секреты,
но сам код шифрования/дешифрования — обычная часть кодовой базы.

Это новый модуль поверх существующего кода. Не трогай основной пайплайн анализа — добавляй
шифрование как отдельный слой до и после пересылки файла.

---

## ВАЖНОЕ ЧЕСТНОЕ ОГРАНИЧЕНИЕ — обязательно отразить в коде и в UI

Поскольку ключ расшифровки передаётся **на тот же инстанс**, где будет происходить и сама
расшифровка, эта схема защищает от:
- кражи файла с диска во время передачи по сети (man-in-the-middle)
- кражи файла, если он случайно останется на диске инстанса после завершения работы
  (зашифрованный файл бесполезен без ключа)
- случайного попадания файла в логи/git/публичный репозиторий

Эта схема **не защищает** от:
- хозяина/оператора арендованной машины, который намеренно инспектирует RAM процесса в
  момент расшифровки (в этот момент сырой текст неизбежно существует в памяти инстанса)
- скомпрометированного образа/темплейта инстанса с встроенным шпионским ПО

**Обязательно добавь это объяснение** как комментарий в коде модуля шифрования (`encryption.py`
или аналогичный) и как текст в UI при первом включении опции "Использовать удалённый GPU" —
короткое, понятное предупреждение, не юридический disclaimer на полстраницы, но честное:
"Шифрование защищает файл при передаче и хранении, но не от оператора арендованного сервера,
который мог бы специально инспектировать память во время обработки. Используйте только
проверенные/высокорейтинговые хосты на Vast.ai, если это для вас критично."

## Архитектура шифрования

### 1. Алгоритм

Используй **AES-256-GCM** (authenticated encryption — даёт и конфиденциальность, и проверку
целостности/отсутствия подмены файла, в отличие от просто AES-CBC). В Python — библиотека
`cryptography` (`from cryptography.hazmat.primitives.ciphers.aead import AESGCM`), это
зрелая, проверенная реализация, не нужно изобретать свою криптографию.

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

def generate_key() -> bytes:
    return AESGCM.generate_key(bit_length=256)

def encrypt_file(plaintext_bytes: bytes, key: bytes) -> bytes:
    aesgcm = AESGCM(key)
    nonce = os.urandom(12)  # 96-bit nonce, стандарт для GCM
    ciphertext = aesgcm.encrypt(nonce, plaintext_bytes, associated_data=None)
    return nonce + ciphertext  # nonce должен путешествовать с шифротекстом

def decrypt_file(encrypted_bytes: bytes, key: bytes) -> bytes:
    nonce, ciphertext = encrypted_bytes[:12], encrypted_bytes[12:]
    aesgcm = AESGCM(key)
    return aesgcm.decrypt(nonce, ciphertext, associated_data=None)
```

### 2. Генерация и передача ключа

1. Ключ генерируется **локально, на машине пользователя**, один раз перед загрузкой файла на
   удалённый инстанс — не на сервере, не заранее захардкожен
2. Ключ кодируется в base64 для удобной передачи (через переменную окружения при запуске
   удалённого процесса, или через защищённый канал SSH — см. ниже способы передачи)
3. **Способ передачи ключа на инстанс** — раз пользователь сознательно выбрал "ключ туда же,
   куда и файл", сделай это максимально просто и явно:
   - Вариант А (проще): ключ передаётся как переменная окружения при старте Docker-контейнера/
     SSH-сессии на инстансе (`CHATSTAT_DECRYPT_KEY=<base64>`), не сохраняется в файл на диске
     инстанса вообще — после расшифровки в памяти, переменная окружения значения не меняет,
     но хотя бы не лежит отдельным файлом, который можно случайно унести в логах/снапшоте диска
   - Вариант Б: ключ передаётся вместе с зашифрованным файлом как отдельный маленький файл
     (`chat.enc` + `chat.key`), который **немедленно удаляется** с диска инстанса сразу после
     успешной расшифровки (`os.remove()` сразу после чтения, до начала анализа)
   - Реализуй вариант А как основной (более чистый), вариант Б как fallback, если по
     какой-то причине передача через env переменную неудобна в конкретном сетапе пользователя

### 3. Процесс на стороне пользователя (локально)

Добавь в существующий backend новый эндпоинт/CLI-утилиту (что удобнее встроить в текущую
архитектуру — например, скрипт `scripts/prepare_remote_upload.py`):

```
Использование:
  python prepare_remote_upload.py --input result.json --output result.enc

Что делает:
1. Читает result.json (или WhatsApp .txt) с диска
2. Генерирует новый случайный ключ AES-256
3. Шифрует содержимое файла
4. Сохраняет result.enc (зашифрованный файл) — это безопасно загружать на удалённый инстанс
5. Выводит ключ в консоль в base64 ОДИН РАЗ, с явной инструкцией:
   "Скопируйте этот ключ — он не сохраняется на диске. Передайте его на удалённый инстанс
   через переменную окружения CHATSTAT_DECRYPT_KEY при запуске обработки."
6. НЕ сохраняет ключ ни в какой файл локально по умолчанию (опционально — флаг
   --save-key-to-file для тех, кто понимает риски и хочет сохранить для повторного использования,
   с явным предупреждением в выводе команды о том, что это менее безопасно)
```

### 4. Процесс на стороне удалённого инстанса (Vast.ai)

В коде, который запускается на арендованном GPU (можно в том же `main.py`, если деплоится
тот же образ, просто с другим режимом запуска, или отдельный минимальный
`remote_worker.py`, если стоит разделить):

1. При старте — читает переменную окружения `CHATSTAT_DECRYPT_KEY` (или файл-ключ, если
   используется вариант Б)
2. Расшифровывает загруженный `.enc` файл **в памяти** (не пишет расшифрованный plaintext
   на диск инстанса вообще, если получится держать весь пайплайн на работе с bytes/DataFrame
   в памяти — структура данных, с которой работает остальной пайплайн (`parser.py` и далее),
   должна уметь принимать байты напрямую, не только путь к файлу на диске)
3. Если **обязательно** нужно записать расшифрованный файл на диск (например, какая-то
   библиотека требует путь к файлу, не байты) — пиши во временную директорию
   (`tempfile.mkdtemp()`), и **гарантированно удаляй** сразу после того, как файл прочитан в
   память (`try/finally` с `os.remove()` в `finally`, чтобы удаление происходило даже при
   ошибке обработки)
4. После завершения анализа — результат (JSON с метриками) **не содержит сырых сообщений**
   в чистом виде, только агрегаты/примеры коротких цитат, необходимые для отображения
   (как уже сделано в существующем пайплайне для разделов типа "Самые яркие моменты") —
   убедись, что итоговый JSON, который скачивается обратно на машину пользователя, не
   содержит избыточно много сырого текста сверх того, что нужно для дашборда

### 5. .gitignore и хранение секретов

1. Добавь в `.gitignore` (если такого паттерна там ещё нет):
   ```
   *.enc
   *.key
   .env.remote
   scripts/generated_keys/
   ```
2. Если ключ для удобства всё же сохраняется локально (флаг `--save-key-to-file` из пункта 3
   выше) — сохраняй в отдельную поддиректорию (`scripts/generated_keys/`), которая полностью
   в `.gitignore`, никогда не в корне репозитория
3. В README — короткий раздел "Безопасная работа с удалённым GPU" с шагами: подготовка
   зашифрованного файла локально → загрузка на Vast.ai инстанс → передача ключа через env
   переменную при запуске → автоматическая расшифровка в памяти на инстансе → получение
   результата обратно. Включи туда же честное предупреждение про ограничения схемы (см. раздел
   выше) без купюр — пользователь должен понимать модель угроз, прежде чем грузить личные
   переписки на чужой сервер.

## Проверка перед сдачей

1. Зашифруй тестовый чат локально, убедись, что `.enc` файл действительно не читается как
   обычный JSON/текст (открой в текстовом редакторе — должен быть бинарный мусор)
2. Передай зашифрованный файл + ключ (через переменную окружения, имитируя удалённый инстанс
   локально для теста) — убедись, что расшифровка проходит успешно и результат совпадает
   побайтово с оригиналом
3. Проверь сценарий повреждённого/подменённого файла — измени один байт в `.enc` файле,
   убедись, что AES-GCM **обнаруживает** подмену (через `InvalidTag` exception при попытке
   расшифровки) — это и есть смысл аутентифицированного шифрования, не просто
   конфиденциальность, но и целостность
4. Убедись, что после расшифровки на "удалённой" стороне временный расшифрованный файл (если
   используется вариант с диском, не чистая работа в памяти) действительно удаляется — проверь
   директорию после завершения обработки
5. Убедись, что `git status` не показывает `.enc`/`.key` файлы как готовые к коммиту после
   обновления `.gitignore`
6. Перечитай добавленное предупреждение про ограничения схемы — убедись, что оно реально
   попадает в README и в UI, не только существует в комментариях кода
